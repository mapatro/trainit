1

How Mule errors work

When a step fails, Mule creates an error with a type (like HTTP:NOT_FOUND or DB:CONNECTIVITY). You catch errors with an error handler, and you can match specific types so different failures get different responses.

2

Try scope: continue vs propagate

Wrap risky steps in a Try scope, which has its own error handler. Inside, two key choices: On Error Continue handles the error and lets the flow finish as a success; On Error Propagate handles it but still re-throws, so the caller knows it failed.

More: error handling (official docs)

3

Handle an error gracefully

● Do this
  1. Add a flow with an HTTP Request to a URL that will fail, e.g. path /this-does-not-exist on the test API.
  2. Wrap the request in a Try scope.
  3. Add an On Error Continue that matches HTTP:NOT_FOUND and Sets Payload to a friendly Sorry, not found message.
  4. Run and trigger it.
4

Test

✓ Check yourself
  • Hitting the failing endpoint returns your friendly message, not a stack trace.
  • Switching to On Error Propagate changes the response into an error the caller sees.
  • You can name two Mule error types from memory.
5

Set a proper status code

Pair error handling with the HTTP response: in the listener's error response, set the status to 404 or 500 so callers get correct HTTP semantics, not a 200 with an error message inside.
