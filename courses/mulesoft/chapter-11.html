1

The shape of every test

An MUnit test has three sections. Behavior sets things up and mocks real systems so tests don't hit live APIs. Execution runs the flow. Validation asserts the result is what you expected. It's the classic given / when / then.

More: MUnit (official docs)

2

Mocks and asserts

mock-when replaces a real connector call with a canned response, so a test for your logic never depends on the internet. assert-that / assert-equals check the payload. Coverage tells you what percentage of your flows the tests exercise.

3

Write a test

● Do this
  1. Create an MUnit Test Suite for your project.
  2. Add a test that runs your /hello flow (Execution).
  3. Add an assert that the payload equals your expected message (Validation).
  4. For a flow with an HTTP Request, add a mock-when so it returns fake data instead of calling the real API.
  5. Run the suite.
4

Test the tests

✓ Check yourself
  • The MUnit suite runs and shows a green pass.
  • Breaking the flow on purpose makes the test go red — proving it actually checks something.
  • You can read the coverage percentage for your app.
5

Why employers care

Tests are how teams trust changes. Being able to say “I write MUnit tests with mocks and asserts” puts you ahead of many junior candidates.
