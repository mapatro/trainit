1

API-led connectivity

MuleSoft organizes APIs into three layers. System APIs unlock raw systems (a database, Salesforce). Process APIs hold business logic and combine data. Experience APIs shape data for a specific channel like a mobile app. The point is reuse: build once, plug in many times.

More: API-led connectivity

2

Design the contract first

You describe an API in a spec language — RAML (MuleSoft's own) or OAS/OpenAPI. The spec lists resources (like /customers), methods, and responses. You can design it in Design Center and share it via Exchange.

3

APIkit turns the spec into flows

Point APIkit at your spec and it scaffolds a router plus an empty flow for every endpoint — the skeleton is generated, you just fill in the logic.

4

Do it

● Do this
  1. Create a simple spec (RAML or OAS) with one resource, e.g. GET /greeting returning a message.
  2. Use the APIkit scaffold action to generate the router and flows.
  3. Implement the GET /greeting flow with a Set Payload.
  4. Run the project.
5

Test

✓ Check yourself
  • The scaffold created one flow per endpoint in your spec.
  • Calling the scaffolded endpoint returns your implemented response.
  • You can explain System vs Process vs Experience APIs in your own words.
